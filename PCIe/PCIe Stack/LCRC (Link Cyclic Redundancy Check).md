
> **LCRC** stands for **Link Cyclic Redundancy Check**. It is an error-detection value generated by the **PCIe Data Link Layer** and attached to every TLP sent across a PCIe link.

Its purpose is:

> **Detect corruption of a TLP while it travels across a single PCIe link.**

## Where LCRC Fits in PCIe

```
Transaction Layer
       |
       v
     TLP
       |
       v
Data Link Layer
       |
       +--> Sequence Number
       +--> LCRC
       |
       v
Physical Layer
```

The Transaction Layer creates the TLP. The Data Link Layer computes and appends the LCRC.

## Why LCRC Is Needed

Suppose the host sends:

```
Memory Write TLP
```

to an NVMe SSD.

During transmission, electrical noise causes:

```
Original:

10110010

Received:

10100010
```

A bit flipped.

Without error detection:

```
Corrupted packet accepted
```

which could corrupt data or registers.

LCRC allows the receiver to detect the corruption.

## What Gets Protected?

LCRC protects the entire TLP transmitted over the link.

Conceptually:

```
+----------------+| TLP Header     |+----------------+| Payload        |+----------------+       |       vCompute CRC       |       vLCRC
```

The LCRC is appended by the Data Link Layer before transmission.

### Transmission Example

Host generates:

```
Memory Write TLP
```

Data Link Layer adds:

```
Seq# = 100

LCRC = 0x12345678
```

Conceptually:

```
+----------------+
| Seq# = 100     |
+----------------+
| TLP            |
+----------------+
| LCRC           |
+----------------+
```

This is sent over PCIe.

### Receiver Side

The SSD receives:

```
Seq# = 100
TLP
LCRC = 0x12345678
```

The SSD computes its own CRC over the received TLP.

```
Received TLP
      |
      v
CRC Generator
      |
      v
Calculated LCRC
```

Then compares:

```
Calculated LCRC
       ==
Received LCRC ?
```

#@ Good Packet

Example:

```
Received LCRC   = 0x12345678
Calculated LCRC = 0x12345678
```

Match.

Result:

```
Packet Valid
```

Receiver sends:

```
ACK DLLP
```

## Corrupted Packet

Example:

```
Received LCRC   = 0x12345678
Calculated LCRC = 0xABCDEF12
```

Mismatch.

Result:

```
Packet Corrupted
```

Receiver sends:

```
NAK DLLP
```

Sender retransmits from replay buffer.

## Relationship to ACK/NAK

LCRC is what allows the receiver to decide:

```
ACK
or
NAK
```

Flow:

```
Receive TLP
      |
      v
Check LCRC
      |
      +--> PASS --> ACK
      |
      +--> FAIL --> NAK
```

f bad:


SSD sends:

```

```

## LCRC vs ECRC

Many engineers confuse these.

### LCRC

**Link CRC**

Generated by:

```
Data Link Layer
```

Purpose:

```
Protect one PCIe link
```

Example:

```
Root Complex <----> NVMe SSD
```

LCRC is checked at every hop.

### ECRC

**End-to-End CRC**

Generated by:

```
Transaction Layer
```

Purpose:

```
Protect transaction end-to-end
```

through switches and bridges.

Much less commonly used.

## LCRC Scope

Suppose:

```
CPU
 |
PCIe Switch
 |
NVMe SSD
```

LCRC is checked separately on:

```
CPU <-> Switch
Switch <-> SSD
```

Each hop generates its own LCRC.

This is why LCRC is called a **link-level CRC**.

## Replay Mechanism

When LCRC fails:

```
TLP #100
```

Receiver:

```
NAK
```

Sender:

```
Replay Buffer
```

contains:

```
#100
#101
#102
```

Sender retransmits. This happens entirely in hardware. Software and NVMe firmware are unaware.

### Real PCIe Gen4 Example

Suppose a Gen4 lane experiences noise.

Host sends:

```
Memory Write TLP
```

One bit flips.

Receiver:

```
LCRC mismatch
```

Receiver sends:

```
NAK DLLP
```

Host replays packet.
Transaction succeeds.
This is normal PCIe operation.

## Why LCRC Is Important

Without LCRC:

```
Corrupted doorbell writesCorrupted DMA transfersCorrupted register accesses
```

could silently occur. LCRC provides the first line of protection against transmission errors.

# Summary

**LCRC (Link CRC)** is a CRC generated by the PCIe Data Link Layer and attached to every transmitted TLP.

Purpose:

```
Detect link-level packet corruption
```

Operation:

```
Sender
  |
Generate LCRC
  |
Transmit TLP
  |
Receiver
  |
Recalculate LCRC
  |
Compare
```

Results:

```
Match     -> ACK
Mismatch  -> NAK + Replay
```

LCRC works together with:

```
Sequence Numbers
Replay Buffers
ACK DLLPs
NAK DLLPs
```

to ensure reliable delivery of PCIe TLPs between devices such as a host and an NVMe SSD.

# How is LCRC generated?

> LCRC (**Link CRC**) is generated by the PCIe **Data Link Layer** using a hardware CRC calculation engine.The purpose is to create a fingerprint of the transmitted TLP so the receiver can detect if any bits changed during transmission.

## High-Level Process

Suppose the Transaction Layer creates a TLP:

```
+----------------------+
| TLP Header           |
+----------------------+
| TLP Data             |
+----------------------+
```

The Data Link Layer does:

```
TLP
 |
 v
CRC Generator
 |
 v
32-bit LCRC
```

Then transmits:

```
+----------------------+
| Sequence Number      |
+----------------------+
| TLP                  |
+----------------------+
| LCRC                 |
+----------------------+
```

## What Is a CRC?

CRC stands for:

```
Cyclic Redundancy Check
```

A CRC is a mathematical function that converts a large block of bits into a fixed-size value.

Example:

```
Input:
1011001000110101...

Output:
0x8A34F2D1
```

If even one bit changes:

```
1011001000110101...
1011001001110101...
```

the CRC result changes dramatically.

## PCIe LCRC Size

PCIe uses:

```
32-bit LCRC
```

So the result is:

```
0x00000000to0xFFFFFFFF
```

## What Data Is Included?

The LCRC is computed over the entire TLP.

Conceptually:

```
+------------------+
| TLP Header       |
+------------------+
| TLP Payload      |
+------------------+
```

The LCRC itself is **not** included in the calculation. The sequence number is also handled separately by the Data Link Layer.

## CRC Polynomial

Internally PCIe uses a specific CRC-32 polynomial.

Conceptually:

```
CRC-32 Polynomial
```

The hardware implements this as a shift-register network. Engineers normally do not calculate it manually.
The PCIe controller contains dedicated CRC logic.

## Hardware Implementation

Inside the PCIe controller:

```
TLP Stream
    |
    v
+-------------+
| CRC Engine  |
+-------------+
    |
    v
LCRC
```

As bits flow through:

```
Bit 0
Bit 1
Bit 2
...
Bit N
```

the CRC engine continuously updates its internal state.

## Simplified Example

Imagine a tiny packet:

```
Header:
1011

Data:
0110
```

Combined:

```
10110110
```

Feed into CRC engine:

```
CRC(10110110)
   |
   v
1101
```

Result:

```
LCRC = 1101
```

Real PCIe uses:

```
32 bits
```

instead of 4 bits.

## Why CRC Instead of Simple Checksum?

Simple checksum:

```
1 + 2 + 3 + 4
```

may miss some errors.

CRC is much stronger at detecting:

```
Single-bit errors
Double-bit errors
Burst errors
Lane noise
Transmission corruption
```

which are common concerns in high-speed PCIe links.

## Practical PCIe Debugging

When engineers see:

```
Replay Counter Increasing
```

often the root cause is:

```
LCRC failures
```

caused by:

- Signal integrity problems
- Connector issues
- Equalization problems
- Excessive noise
- Marginal Gen4/Gen5 links

# Summary

LCRC is generated by a hardware **CRC-32 engine** in the PCIe Data Link Layer.

Process:

```
TLP
 |
 v
CRC-32 Calculation
 |
 v
32-bit LCRC
 |
 v
Append to Packet
```

Receiver performs the same calculation:

```
Received TLP
 |
 v
CRC-32 Calculation
 |
Compare
```

Results:

```
Match     -> ACK
Mismatch  -> NAK + Replay
```

This mechanism allows PCIe to detect corrupted TLPs and automatically recover them without involving software or NVMe firmware.
