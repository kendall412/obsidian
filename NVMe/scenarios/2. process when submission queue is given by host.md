
When the **host gives a Submission Queue entry to an NVMe controller**, the process is:

```
Host software
   ↓
Build NVMe command in Submission Queue
   ↓
Ring Submission Queue Doorbell
   ↓
NVMe controller fetches command
   ↓
Controller performs command
   ↓
Controller writes Completion Queue entry
   ↓
Controller interrupts host or host polls CQ
   ↓
Host processes completion
   ↓
Host rings Completion Queue Doorbell
```

## 1. Host creates Submission Queue and Completion Queue

Before commands can be submitted, the host must create queues.

For the **Admin Queue**, this is done during controller initialization using registers:

```
AQA   = Admin Queue Attributes
ASQ   = Admin Submission Queue base address
ACQ   = Admin Completion Queue base address
CC.EN = 1 to enable controller
```

For **I/O Queues**, the host sends Admin commands:

```
Create I/O Completion Queue
Create I/O Submission Queue
```

A typical pair looks like:

```
Submission Queue 1  → Completion Queue 1
Submission Queue 2  → Completion Queue 2
...
```

Each Submission Queue contains fixed-size command entries.

```
One NVMe Submission Queue Entry = 64 bytes
```

Each Completion Queue contains fixed-size completion entries.

```
One NVMe Completion Queue Entry = 16 bytes
```

## 2. Host prepares data buffer in memory

For commands that transfer data, such as **Read** or **Write**, the host prepares a memory buffer.

Example for a Read:

```
Host wants to read LBA 1000 into RAM address 0x80000000
```

The host buffer may be described using:

```
PRP = Physical Region PageSGL = Scatter Gather List
```

Commonly, NVMe uses PRPs.

Example:

```
PRP1 = physical address of first data buffer pagePRP2 = physical address of second page or PRP list
```

The NVMe controller will later use PCIe DMA to read or write this host memory.

## 3. Host builds the Submission Queue Entry

The host writes a 64-byte command into the Submission Queue memory.

A simplified NVMe Read command looks like this:

```
DW0      : OPC, FUSE, PSDT, CID
NSID     : Namespace ID
MPTR     : Metadata Pointer
PRP1/SGL : Data buffer pointer
PRP2     : Data buffer pointer or PRP list pointer
CDW10    : Starting LBA low
CDW11    : Starting LBA high
CDW12    : Number of logical blocks
CDW13-15 : Command-specific fields
```

Important fields:

| Field     | Meaning                               |
| --------- | ------------------------------------- |
| OPC       | Opcode, such as Read, Write, Identify |
| CID       | Command Identifier                    |
| NSID      | Namespace ID                          |
| PRP1/PRP2 | Host memory buffer addresses          |
| SLBA      | Starting Logical Block Address        |
| NLB       | Number of logical blocks to transfer  |

Example:

```
OPC  = 0x02        ; Read
CID  = 0x1234
NSID = 1
SLBA = 1000
NLB  = 7           ; means 8 logical blocks
PRP1 = 0x80000000
```

Important detail:

```
NLB is zero-based.
NLB = 0 means 1 block.
NLB = 7 means 8 blocks.
```

## 4. Host updates its Submission Queue Tail Pointer

The host maintains a local software tail pointer for the Submission Queue.

Example:

```
SQ size = 8 entries

Before submission:
SQ Tail = 0

Host writes command at SQ[0]

After submission:
SQ Tail = 1
```

The host must not overwrite commands that have not yet completed.

Conceptually:

```
SQ Head = owned/advanced by controller
SQ Tail = owned/advanced by host
```

The queue is circular:

```
SQ[0] → SQ[1] → SQ[2] → ... → SQ[7] → SQ[0]
```

## 5. Host rings the Submission Queue Doorbell

After writing the command into memory, the host writes the new tail value to the Submission Queue Tail Doorbell register.

```
SQ Tail Doorbell = new SQ Tail
```

Example:

```
Host writes 1 to SQ1TDBL
```

This tells the controller:

```
“There is a new command available in Submission Queue 1.”
```

Important:

The host must make sure the command is visible in memory before ringing the doorbell.

So the order is:

```
1. Write command into SQ memory
2. Memory barrier / ordering
3. Write SQ Tail Doorbell

Not:

1. Ring doorbell
2. Write command
```

## 6. Controller sees the doorbell update

The NVMe controller monitors its doorbell registers through PCIe MMIO.

When it observes the SQ Tail Doorbell change, it knows that new commands are available between:

```
Controller's SQ Head
        and
Host's SQ Tail
```

Example:

```
Controller SQ Head = 0
Host SQ Tail       = 1
```

So the controller knows:

```
SQ[0] contains a new command
```


# 7. Controller fetches the command from host memory

The Submission Queue itself is in **host memory**, not usually inside the controller.

The controller fetches the SQE using PCIe DMA.

```
Controller issues PCIe Memory Read
        ↓
Reads 64-byte SQE from host memory
        ↓
Decodes command
```

Example:

```
Fetch SQ[0]
Decode:
  OPC  = Read
  CID  = 0x1234
  NSID = 1
  SLBA = 1000
  NLB  = 7
  PRP1 = 0x80000000
```

## 8. Controller validates the command

The controller checks whether the command is valid.

It may validate:

```
Opcode supported?
Queue valid?
Namespace valid?
LBA range valid?
PRP/SGL valid?
Command fields legal?
Controller state ready?
```

If the command is invalid, the controller does not perform the requested operation. Instead, it posts a completion with an error status.

Example:

```
Status Code Type = Generic Command Status
Status Code      = Invalid Field in Command
```

## 9. Controller executes the command

For a Read command, the controller performs roughly this flow:

```
1. Translate LBA to NAND/flash physical location
2. Read data from NAND or cache
3. Perform ECC/error correction
4. Transfer data to host memory using PCIe DMA write
5. Post completion
```

For a Write command:

```
1. Fetch data from host memory using PCIe DMA read
2. Store data in controller buffer
3. Program data to NAND or cache it
4. Update FTL mapping
5. Post completion
```

For an Admin command, such as Identify:

```
1. Decode Admin opcode
2. Gather requested controller or namespace data
3. DMA result structure into host memory
4. Post completion
```

## 10. Controller advances its internal Submission Queue Head

After the controller consumes the command, it advances its internal SQ Head pointer.

Example:

```
Before:
SQ Head = 0

After consuming SQ[0]:
SQ Head = 1
```

The host does **not** directly write the Submission Queue Head.

The controller reports the SQ Head value inside the Completion Queue Entry.


## 11. Controller writes a Completion Queue Entry

When the command finishes, the controller writes a 16-byte Completion Queue Entry into the associated Completion Queue.

A Completion Queue Entry includes:

```
DW0 : Command-specific resultDW1 : Reserved or command-specificDW2 : SQ Head Pointer, SQ IdentifierDW3 : Command Identifier, Phase Tag, Status
```

Important fields:

|Field|Meaning|
|---|---|
|CID|Matches the original command|
|SQID|Submission Queue ID|
|SQHD|New Submission Queue Head Pointer|
|SCT|Status Code Type|
|SC|Status Code|
|P|Phase Tag|

Example completion:

```
CID  = 0x1234
SQID = 1
SQHD = 1
SCT  = 0
SC   = 0
P    = 1
```


This means:

```
Command 0x1234 from SQ1 completed successfully.
Controller has consumed SQ entries up to head pointer 1.
```

## 12. Controller updates Completion Queue Tail internally

For Completion Queues:

```
CQ Head = owned/advanced by host
CQ Tail = owned/advanced by controller
```

The controller writes the completion entry at its current CQ Tail position, then advances its internal CQ Tail.

Example:

```
CQ Tail = 0

Controller writes completion at CQ[0]

CQ Tail = 1
```

The host discovers new completion entries by checking the **Phase Tag** or by receiving an interrupt.

## 13. Controller generates interrupt, unless interrupts are disabled

If interrupts are enabled, the controller sends an interrupt to the host.

Usually this is done through:

```
MSI-X interrupt
```

The interrupt tells the host:

```
“Completion Queue has new entries.”
```

In high-performance systems, the host may use polling instead of interrupts.

Polling means the host repeatedly checks the Completion Queue memory.

## 14. Host processes the Completion Queue Entry

The host reads the Completion Queue Entry from host memory.

It checks:

```
Phase Tag
CID
SQID
Status Code Type
Status Code
```

For success:

```
SCT = 0
SC  = 0
```

For failure, the host checks the status code.

Example:

```
SCT = 0x0
SC  = 0x02
```

Could mean:

```
Invalid Field in Command
```

The host uses the CID to match the completion back to the original command.

```
CID 0x1234 completed
```

## 15. Host advances Completion Queue Head

After processing the completion, the host advances its local Completion Queue Head pointer.

Example:

```
Before:
CQ Head = 0

Host processes CQ[0]

After:
CQ Head = 1
```

Then the host writes the new value to the Completion Queue Head Doorbell register.

```
CQ Head Doorbell = 1
```

This tells the controller:

```
“The host has processed completion entries up to this point.”
```

## 16. Controller now knows CQ entries are free

Once the host rings the Completion Queue Head Doorbell, the controller knows it may reuse those Completion Queue slots.

Ownership is:

```
Submission Queue:  Host produces entries  Controller consumes entriesCompletion Queue:  Controller produces entries  Host consumes entries
```

# Complete Read Command Example

Assume:

```
SQID = 1
CQID = 1
SQ size = 8
CQ size = 8
Host SQ Tail = 0
Controller SQ Head = 0
Host CQ Head = 0
Controller CQ Tail = 0
```

Host wants to read 8 logical blocks from LBA 1000.

## Step A: Host writes SQE

```
SQ[0]:
  OPC  = Read
  CID  = 0x55
  NSID = 1
  SLBA = 1000
  NLB  = 7
  PRP1 = 0x80000000
```

## Step B: Host rings SQ doorbell

```
SQ Tail = 1
write SQ1TDBL = 1
```

## Step C: Controller fetches command

```
Controller reads SQ[0] by DMA
Controller decodes Read command
```

## Step D: Controller transfers data

```
Controller reads NAND/cache
Controller DMA-writes data to host memory at 0x80000000
```

## Step E: Controller posts completion

```
CQ[0]:
  CID  = 0x55
  SQID = 1
  SQHD = 1
  SC   = 0
  SCT  = 0
  P    = 1
```

## Step F: Host handles completion

```
Host reads CQ[0]
Host sees CID 0x55 completed successfully
Host advances CQ Head to 1
Host writes CQ1HDBL = 1
```

## Doorbell direction summary

|Doorbell|Written by|Purpose|
|---|---|---|
|SQ Tail Doorbell|Host|Tell controller new commands are available|
|CQ Head Doorbell|Host|Tell controller completions have been consumed|

The controller does not usually ring doorbells. It updates queue memory and may generate interrupts.

## Pointer ownership summary

|Queue|Producer|Consumer|Producer pointer|Consumer pointer|
|---|---|---|---|---|
|Submission Queue|Host|Controller|SQ Tail|SQ Head|
|Completion Queue|Controller|Host|CQ Tail|CQ Head|

## Very important concept

The Submission Queue is not “sent” directly to the NVMe device command-by-command.

Instead:

```
Host places commands in host memory
Host rings doorbell
Controller fetches commands using PCIe DMA
Controller executes commands
Controller writes completions back to host memory
Host processes completions
```

That is the central NVMe queue mechanism.
